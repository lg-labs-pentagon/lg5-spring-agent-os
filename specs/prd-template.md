# PRD — <feature or service name>

> Replace every `<...>` placeholder. Delete sections that don't apply with a
> `_(N/A: <reason>)_` note instead of removing them silently.

## 1. Summary

One paragraph (≤5 sentences) describing what is being built and why now.
The reader should be able to decide if they care about the rest of this
document from this paragraph alone.

## 2. Problem

What problem does this solve? Frame it from the user's perspective, not
ours. Include:

- Who experiences the problem.
- How they experience it today (current workaround or absence of
  capability).
- What goes wrong if we don't solve it.

## 3. Users / personas

Who are the primary users of this capability? For each:

- Persona name.
- Their current relationship to the system.
- Their success metric (how they know this helps them).

## 4. Success metrics

Measurable outcomes that determine whether the feature succeeded. Prefer
**leading** indicators (adoption, usage frequency) over lagging ones
(revenue) when leading is sufficient signal.

| Metric | Baseline | Target | Measurement window |
|--------|---------:|-------:|--------------------|
| <m1>   | <today>  | <goal> | <e.g. 30 days post-launch> |

## 5. Scope (in)

The minimum useful subset. Each item should be testable and shippable.

- [ ] <capability 1>
- [ ] <capability 2>
- [ ] …

## 6. Scope (out)

Explicitly NOT in scope, with rationale. This protects the team from scope
creep and signals to stakeholders what to expect later.

- <thing> — _(reason: <why deferred>)_
- <thing> — _(reason: …)_

## 7. Architecture / dependencies

Cross-references the architecture decisions:

- New service? Identifies the 8-module shape per RULE-004.
- Saga participant? Lists the steps and the orchestrator (use the
  `lg5-saga` skill).
- Outbox events emitted? List by event name (use `/add-outbox` per event).
- Kafka topics consumed/produced? List by name + Avro schema.
- External systems integrated? Feign clients per RULE-014 prefix.

| Dependency | Type | Owner | Status |
|-----------|------|-------|--------|
| <s1>      | upstream-svc | <team> | <ready / blocked> |

## 8. Acceptance criteria

Concrete, runnable checks that demonstrate the feature works:

- [ ] `make install-skip-test` succeeds.
- [ ] `make run-acceptance-test` includes a green scenario for <happy path>.
- [ ] `make run-acceptance-test` includes a green scenario for <failure /
  rollback path> (RULE-009, RULE-010).
- [ ] No `must` violations from `lg5-code-reviewer` subagent.
- [ ] PRD-specific behavioral assertions: <list>.

## 9. Open questions

Decisions deferred to specific people / dates.

| Question | Decider | Due |
|---------|---------|-----|
| <q1>    | <name>  | <date> |

## 10. Implementation plan reference

Generated by the `lg5-planner` subagent. Either inline the plan here or
reference its file path.

plan-output-from-lg5-planner.md
